## 什么是 Superpowers

**Superpowers** 是由 Jesse Vincent ([@obra](https://github.com/obra)) 创建的 AI 编程工作流框架（GitHub 65k+ stars）。它不是 VS Code 的传统扩展，而是一套为 AI 编程助手提供**结构化工作流**的"技能包"。

### 核心理念

```
传统 AI 编程：                   Superpowers：
用户 → 手动描述需求              用户 → 选择工作流
      → AI 自由发挥                    → AI 按照结构化的步骤执行
      → 结果随机                       → 结果可预测、可重复
```

**Superpowers 让 AI 从"自由聊天"升级为"结构化协作"。**

### 提供的核心工作流

| 命令 | 用途 | 难度 |
|------|------|------|
| `/plan` | 制定实施计划 | ⭐ |
| `/tdd` | 测试驱动开发 | ⭐⭐ |
| `/debug` | 系统化 Debug | ⭐ |
| `/review` | 代码审查 | ⭐ |
| `/verify` | 验证完成度 | ⭐ |
| `/finish` | 分支清理 | ⭐⭐ |
| `/explain` | 解释代码 | ⭐ |
| `/refactor` | 重构 | ⭐⭐ |
| `/brainstorming` | 头脑风暴 | ⭐ |
| `/help` | 查看帮助 | ⭐ |

## 安装 Superpowers

### 前置条件

- **VS Code** 1.90.0 或更高版本
- **GitHub Copilot** 扩展（Chat 激活）
- 或 **Claude Code for VS Code** 扩展

### 方式一：通过 GitHub Copilot 安装（推荐）

1. 在 VS Code 中打开 Chat 面板（`Cmd+Shift+I`）
2. 在 Chat 输入框中点击设置 → **Plugins**
3. 点击 **"从源安装插件"**
4. 输入插件源地址：
   ```
   https://github.com/obra/superpowers
   ```
5. 安装成功后，在 Chat 中输入 `@superpowers /help` 验证

### 方式二：通过命令行安装

```bash
# 在 VS Code 终端或 Copilot CLI 中执行
/plugin marketplace add obra/superpowers-marketplace
/plugin install superpowers@superpowers-marketplace
```

### 方式三：通过 Superpowers 扩展

也可以直接安装 VS Code 扩展：

```bash
ext install superpowers-for-copilot-chat dwaintr.superpowers-vscode
```

安装后，在 Chat 中使用 `@superpowers` 参与者调用命令。

### 验证安装

在 GitHub Copilot Chat 中输入：

```
@superpowers /help
```

如果返回可用命令列表，安装成功！

## 核心工作流详解

### `/plan` — 计划先行

在写任何代码之前，让 AI 先帮你制定完整的实施计划。

```
@superpowers /plan 给博客添加标签分类功能

需要支持：
- 文章可以添加多个标签
- 标签页面显示所有标签
- 点击标签过滤文章
```

**工作流输出的计划：**
```
📋 实施计划
├── 1. 数据库：创建 tags 表 + post_tags 关联表
├── 2. 后端：Tag CRUD API + 文章标签关联 API
├── 3. 前端：标签组件（LabelSelector、TagList）
├── 4. 页面：标签索引页、标签文章列表页
├── 5. 测试：单元测试 + 集成测试
└── 6. 文档：更新 README
```

**工作原理：**
1. AI 分析你的需求
2. 生成结构化的实施步骤（可编辑）
3. 你确认或修改计划
4. 之后可以逐一执行

### `/tdd` — 测试驱动开发

按照"先写测试，再写实现"的 TDD 最佳实践工作流。

```
@superpowers /tdd 实现一个防抖函数
```

**工作流：**
```
🟢 步骤 1/3: 编写测试
- 写测试用例（正常情况、边界值、异步场景）
- 确认测试（会失败）        ← AI 在这里等你确认

🟡 步骤 2/3: 编写实现
- 编写使测试通过的代码
- 优化实现

🔵 步骤 3/3: 验证
- 运行测试确认全部通过
- 代码审查
```

### `/debug` — 系统化调试

不再漫无目的地加 `console.log`。

```
@superpowers /debug 用户登录后 token 存储正常，但刷新页面后 token 丢失
```

**工作流：**
```
🔍 分析阶段
├── 假设 1: localStorage 写入失败
├── 假设 2: 读取时的 key 不匹配
├── 假设 3: 应用初始化时覆盖了 token
└── 假设 4: 过期时间计算错误

🛠 验证阶段
└── 逐条排除假设...

✅ 修复阶段
└── 执行修复 + 验证
```

### `/review` — 代码审查

在提交 PR 之前，让 AI 先审查你的代码。

```
@superpowers /review

请审查 src/auth/ 目录下的所有变更
```

**审查维度：**
- **安全性**：SQL 注入、XSS、认证漏洞
- **性能**：循环效率、内存泄漏、不必要的请求
- **可维护性**：命名、结构、重复代码
- **正确性**：边界条件、错误处理

### `/verify` — 验证完成

确认你的代码变更确实完成了所声明的功能。

```
@superpowers /verify

检查用户注册功能是否完整实现：
- 表单验证
- 密码加密
- 邮箱唯一性检查
- 发送验证邮件
```

### `/finish` — 分支完成

在完成一个功能分支时，自动进行清理工作。

```
@superpowers /finish
```

**自动执行：**
1. 运行测试套件
2. 检查未提交的变更
3. 审查代码质量
4. 清理临时文件
5. 建议 PR 描述

## Superpowers 配合 VS Code 使用

### 高效工作流推荐

```
日常开发流程：

1. /plan           → 接到新需求先做计划
2. /tdd            → 测试驱动实现每个功能
3. /review         → 提交前自我审查
4. /verify         → 确认全部功能正常
5. /finish         → 清理并创建 PR
```

### 与 VS Code 功能联动

| VS Code 功能 | Superpowers 对应 |
|-------------|-----------------|
| 命令面板 `Cmd+Shift+P` | `@superpowers /command` |
| 终端 `Ctrl+\`` | `/debug` 自动执行测试 |
| 源代码管理 `Cmd+Shift+G` | `/finish` 自动 Git 操作 |
| 问题面板 `Cmd+Shift+M` | `/review` 结果自动输出 |
| 调试器 `F5` | `/debug` 系统化分析 |

## 编写自定义 Superpowers

如果你想创建自己的工作流：

### 1. 创建工作流文件

```javascript
// .github/copilot/scripts/my-workflow.md
## My Custom Workflow

这是我自定义的工作流，用于自动化处理日常任务。

### 步骤

我需要你帮我完成以下操作：
1. 分析当前项目的代码结构
2. 找出所有 TODO 注释
3. 按优先级排序
4. 生成一份报告
```

### 2. 注册工作流

在项目的 Copilot 配置中添加：

```jsonc
// .github/copilot/scripts/config.json
{
  "scripts": [
    {
      "name": "my-workflow",
      "description": "我的自定义工作流",
      "prompt": "运行我的自定义工作流"
    }
  ]
}
```

### 3. 调用

```
@superpowers /my-workflow
```

## 最佳实践

### ✅ 正确使用

```
✅ 好的用法：
- 在开始编码前用 /plan 制定计划
- 复杂逻辑用 /tdd 确保测试覆盖
- 提交前用 /review 自我审查
- 遇到 Bug 用 /debug 系统化排查

❌ 不好的用法：
- 对简单的变量重命名使用 /plan（直接 F2 更快）
- 写死代码后全指望 /review 来发现问题
- 在 AI 无法访问上下文时仍然依赖它
```

### 适用场景速查

| 场景 | 推荐工作流 |
|------|-----------|
| 新功能开发 | `/plan` → 逐个 `/tdd` |
| 修复 Bug | `/debug` → `/verify` |
| 代码质量提升 | `/review` → `/refactor` |
| 技术预研 | `/brainstorming` → `/plan` |
| 准备提交 | `/review` → `/finish` |
| 学习新代码库 | `/explain` |

### 常用提示

1. **提供上下文**：在请求中包含相关的文件路径、代码片段或错误信息
2. **明确期望**：告诉 AI 你想要什么产出（计划、代码、解释）
3. **迭代使用**：先用 `/plan` 对齐方向，再用 `/tdd` 逐步实现
4. **组合工作流**：例：`/plan` → `/tdd` → `/review` → `/finish`

## 进阶技巧

### 结合自定义指令

在项目根目录创建 `.github/copilot-instructions.md`：

```markdown
## 项目技术栈
- 前端: React + TypeScript + Tailwind CSS
- 后端: Node.js + Express
- 数据库: PostgreSQL
- 测试: Vitest

## 编码规范
- 使用函数组件和 Hooks 而非类组件
- 所有 API 响应使用统一格式 { success, data, error }
- 测试覆盖率目标 ≥ 80%
```

### 多文件提示

```
@superpowers /refactor 

请将 src/utils/helpers.ts 中的函数按职责拆分到：
- src/utils/format.ts
- src/utils/validate.ts
- src/utils/api.ts

保持导出接口兼容。
```
